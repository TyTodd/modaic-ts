# Plan: implement `src/serialization/config.ts::serializeSignatureToConfig`

Goal: fill in the existing stub so a `Signature` (Zod-based) authored in **modaic-ts**
serializes into the same `config.json` the Python SDK writes, so a TS-created Arbiter is
deserializable by the Python server/SDK. One-directional only (TS → Python); we never need
to read Python config.json back into a TS `Signature`.

## Current state (already wired)

- `src/serialization/config.ts` exports a stub `serializeSignatureToConfig(signature, title): ConfigJson`
  that throws "not implemented yet". `ConfigField` / `ConfigJson` interfaces already exist.
- `Arbiter.create` / `Arbiter.update` (`src/arbiter/arbiter.ts`) already call it with
  `title = repoNameToTitle(repo)`, then push `{ config, program }` via git. **No caller changes needed.**
- `buildProgramJson` (program.json) is already done and tested; this is the missing sibling.

## Golden reference

`core/modaic/tests/artifacts/temp/modaic_cache/sync/TyTodd/predict-test-repo/config.json`:

```json
{
  "model": "openai/gpt-4o-mini",
  "signature": {
    "description": "Summarize the given text into a concise summary.",
    "properties": {
      "text":    { "__dspy_field_type": "input",  "desc": "The text to summarize",      "prefix": "Text:",    "title": "Text",    "type": "string" },
      "summary": { "__dspy_field_type": "output", "desc": "A concise summary of the text", "prefix": "Summary:", "title": "Summary", "type": "string" }
    },
    "required": ["text", "summary"],
    "title": "SummarizeSignature",
    "type": "object"
  }
}
```

- Envelope is `{model, signature}`. The TS Arbiter has no local LM, so **`model: null`** (matches the stub type).
- The `signature` sub-tree is exactly `dspy.Signature.model_json_schema(...)` (see Python `serializers.py::serialize_signature`).
- Keys are **alphabetically sorted** at every level except `properties` (field order = inputs then outputs). Worth preserving: these files are committed to the hub via git, so stable key order = clean diffs.

## What modaic-ts makes easy (vs the general dspy case)

- **`__dspy_field_type` is free**: a field is `"input"` if it came from `signature.input.shape`, `"output"` if from `signature.output.shape`. No per-field metadata to read.
- **`title` (top-level) is passed in** as the `title` arg — no need to invent a name source.
- **`instructions`** is always populated on the Signature (auto-generated default matches dspy's wording) → top-level `description`.
- **`prefix`** is already computed: `signature.prefixes[name]` = `infer_prefix(name) + ":"`.

## What MUST be correct for Python deserialization (vs what's only cosmetic)

Verified against `serializers.py::_deserialize_dspy_signatures`:

- **Required**: top-level `title` (non-empty string — `make_signature` crashes on `None`); `properties` in field order; per-field `__dspy_field_type` (`"input"` else treated as output); the **type fragment** keys `json_to_type` understands; per-field `default` only when present (this, not `required`, drives optionality). `desc` is read (and used in prompts) but optional.
- **Ignored by the deserializer** (we still emit them for byte-parity + clean git diffs): per-field `prefix` (regenerated from the name Python-side), per-field `title` (regenerated by pydantic), the `required` array (Python derives required-ness from `default`).

So byte-parity is a *nice-to-have for diffs/golden tests*; correctness only needs the first bullet.

## Implementation

Put the assembler in `config.ts`; factor the Zod walker + helpers into a sibling
`src/serialization/zod_schema.ts` (keeps `config.ts` readable). Steps inside `serializeSignatureToConfig`:

1. **Walk fields**: iterate `signature.input.shape` (→ `__dspy_field_type: "input"`) then
   `signature.output.shape` (→ `"output"`), preserving insertion order.
2. **Per field**, build the object:
   - `__dspy_field_type`
   - `desc`: reuse `dump_state`'s precedence + dspy default →
     `meta()?.description ?? meta()?.desc ?? schema.description ?? "${" + name + "}"`
     (`.describe("x")` is read via `.meta().description` in zod 4 — `_def.description` is undefined; verified).
   - `prefix`: `signature.prefixes[name]`.
   - `title`: `pydanticTitle(name)` — **NOT** `infer_prefix`. Mirrors Python `name.replace("_"," ")` then `str.title()`:
     `name.replace(/_/g," ").replace(/[A-Za-z]+/g, w => w[0].toUpperCase()+w.slice(1).toLowerCase())`.
     (`text`→"Text", `my_field`→"My Field", `HTMLParser`→"Htmlparser", `text2number`→"Text2Number".)
     Omit `title` when the field's type fragment is a pure `$ref` (matches pydantic).
   - `default`: include only if the Zod field has a default (`_def.type === "default"` → `_def.defaultValue`).
   - spread the **type fragment** from the walker.
3. **`required`**: field names with no default (and not bare `.optional()`), inputs-then-outputs order.
4. **Assemble** `signature = { description: signature.instructions, properties, required, title, type: "object", ...( $defs if any ) }`.
5. **Sort** keys recursively to alphabetical, skipping `properties` children and `default` payloads
   (`$` < `_` < letters, so `$defs`/`$ref`/`__dspy_field_type` sort first — matches pydantic).
6. Return `{ model: null, signature }`.

### Zod → type-fragment table (the walker)

| Field (Zod) | Fragment |
|---|---|
| `z.string()` | `{"type":"string"}` |
| `z.number()` | `{"type":"number"}` |
| `z.int()` | `{"type":"integer"}` (detect via `_def.format === "safeint"`) |
| `z.boolean()` | `{"type":"boolean"}` |
| `z.array(T)` | `{"type":"array","items":<T or {}>}` |
| `z.enum([...])` (strings) | `{"enum":[...],"type":"string"}` |
| `z.literal("x")` | `{"const":"x","type":"string"}` (keeps `type`) |
| int literals / int enum | `{"enum":[...],"type":"integer"}` |
| `z.T.nullable()` | `{"anyOf":[<T>,{"type":"null"}]}` (+ field `title`, + `default` if any) |
| `z.T.optional()` | unwrap to `<T>`; mark **not required**; no `default` key |
| `z.T.default(v)` | `<T>` + field-level `default: v`; not required |
| `z.record(z.string(), V)` | `{"type":"object","additionalProperties":<V or true>}` |
| `z.tuple([A,B])` | `{"type":"array","maxItems":2,"minItems":2,"prefixItems":[<A>,<B>]}` |
| `z.object({...})` (named) | field: `{"$ref":"#/$defs/Name"}` (no `title`); def registered in `$defs` |
| `modaic.Scale(lo,hi)` | `{"enum":[lo..hi],"type":"integer"}` (see Special types) |
| `modaic.Enum(...v)` | `{"enum":[...v],"type":"string"}` (or `const` for one value) |
| `modaic.Image()` / `Audio()` | field: `{"$ref":"#/$defs/Image"}` (no `title`); `$defs["Image"]={"type":"dspy.Image"}` |

Discriminator: read `schema._def.type` (`string|number|boolean|array|enum|literal|nullable|optional|default|object|record|tuple|union`); unwrap `nullable/optional/default` via `_def.innerType`.

### Widen the interfaces

`ConfigField` currently only allows `type: string`. Generalize it to carry the full JSON-Schema
key set (optional `items`, `enum`, `const`, `anyOf`, `$ref`, `additionalProperties`, `prefixItems`,
`minItems`, `maxItems`, `uniqueItems`, `default`, optional `title`), and add an optional
`$defs?: Record<string, unknown>` to `ConfigJson["signature"]`. Keep `model: null`.

## Special types: `Image` / `Audio` / `Scale` / `Enum` (modaic-ts)

Add `src/signatures/types.ts` exporting friendly factories, mirroring Python `modaic/types.py`
(which already has `Scale` and `Enum`). **Verified key result: none of these require Python or
server changes** — the server deserializes via the modaic SDK, and each of these reduces to
something that SDK already accepts.

How each one serializes (confirmed by running `serialize_signature` on the Python equivalents):

- **`Scale(lo, hi)`** → emits a plain **integer enum**: `{"enum":[lo, lo+1, …, hi], "type":"integer"}`.
  Python `modaic.Scale[1,5]` already serializes this exact way (its `__get_pydantic_json_schema__`
  delegates to `Literal[1..5]`) and round-trips to `Literal[1,2,3,4,5]`. No `$defs`, no tag.
- **`Enum(...values)`** → plain **string enum**: `{"enum":[...], "type":"string"}` (`const` for a single
  value). Same as Python `modaic.Enum[...]`. The noisy-output normalization that Python's Enum/Scale do
  is *server-side validation only* — irrelevant to TS authoring (arbiters never run an LLM locally), so
  the TS versions are thin schema wrappers.
- **`Image` / `Audio`** → these are **real TS classes** in modaic-ts that play the exact role
  `dspy.Image` / `dspy.Audio` play in the Python SDK: a way to declare a multimodal input/output field.
  The class is the value holder (mirrors `dspy.Image`'s `url`/data); a companion **tagged Zod schema** is
  what goes in the signature's `z.object`. The field serializes to the dspy custom-type shape — field
  `{"$ref":"#/$defs/Image", …}` (no `title`, no `type`) plus `$defs["Image"] = {"type":"dspy.Image"}`
  (`$defs` key = the class name `"Image"`/`"Audio"`). **The wire tag stays `dspy.Image` / `dspy.Audio`** so
  the SDK's existing `DSPY_CUSTOM_TYPES` registry reconstructs them into real `dspy.Image` / `dspy.Audio`
  (which dspy's adapters already render as multimodal content). `modaic.Image` is purely the TS-facing
  name; only the serialized tag is `dspy.*`. This is why the signature walker **must** detect them — an
  image field is not a structural object to recurse into, it's an opaque tagged ref.

**Representation + detection.** Two kinds of special types, both detected the same way:
- `Image` / `Audio` are classes; the signature field is a tagged Zod schema, e.g.
  `z.custom<Image>().meta({ __modaic_type: "image" })` (a small `Image.field()` / `imageField()` helper
  hides this). Value construction (`new Image({ url })`) is separate from the signature schema.
- `Scale(lo,hi)` / `Enum(...)` are not classes — just helpers returning a tagged Zod schema
  (`.meta({ __modaic_type: "scale", lo, hi })`; Enum can be plain `z.enum([...])`).

The walker checks `schema.meta()?.__modaic_type` **first**; if set, it emits the special fragment above
(ref+def for image/audio, enum for scale/enum); otherwise it falls through to structural Zod walking.
Detection stays robust and localized, and the class name supplies the `$defs` key.

**Why not a real `modaic.Image` wire tag?** Emitting `{"type":"modaic.Image"}` would make
`_handle_custom_type` raise `ValueError("Invalid type")` unless we also patch `serializers.py`
(`MODAIC_CUSTOM_TYPES` + define Python `modaic.Image`). Reusing the `dspy.*` tag is zero-change and
already what `modaic.Scale`/`Enum` do in spirit (reduce to a type dspy understands). Recommend reusing
`dspy.*` tags for Image/Audio. If true `modaic.*`-branded tags are ever wanted, that's a separate,
explicitly cross-repo change (modaic-sdk + server adapter).

## Decisions / divergences (call out, mostly edge cases for pure-judge arbiters)

1. **Scope of types**: real arbiter signatures are primitive (string/number/bool/enum/optional).
   Implement those first; `array`/`record`/`tuple`/nested-`$defs` are secondary.
2. **Nested `z.object` `$defs`**: needs a name (Zod objects are anonymous → use `.meta({id})` or skip).
   Defs won't carry `__dspy_field_type` (no input/output notion inside) — functionally safe
   (`_handle_custom_type` ignores it). Low priority.
3. **`.optional()` without `.default()`** → omitted from `required`, no `default` key (Python would
   re-read it as required since there's no `default`). Prefer `.default()` to express optionality.
4. **`model`** stays `null` (no client-side LM). The golden's `"openai/gpt-4o-mini"` is just because
   that Python Predict had an explicit LM; not something TS reproduces.
5. **`title` parity**: the golden title `SummarizeSignature` comes from the Python class name. In
   production TS derives it from the repo via `repoNameToTitle`. For the golden test, pass
   `title = "SummarizeSignature"` explicitly to get an exact match.

## Testing (`tests/serialization/config.test.ts`, bun)

1. **Golden parity** (mirrors `program.test.ts`): build the `text -> summary` Signature with the same
   instructions + `.describe()`s, call `serializeSignatureToConfig(sig, "SummarizeSignature")`, and
   `expect(result.signature).toEqual(<golden.signature>)` and `expect(result.model).toBeNull()`.
   (`toEqual` is order-insensitive; add a `JSON.stringify` key-order assertion if we want to lock byte order.)
2. **Type coverage**: small signatures exercising number/int/boolean/enum/literal/optional/default/array
   **and the special types** (`Scale(1,5)` → int enum, `Enum(...)` → string enum, `Image()`/`Audio()` →
   `$ref` + `$defs` with `dspy.*` tag), asserting each field fragment matches the Python output (capture
   goldens by running `serialize_signature` in `core/modaic` like the program.json reference was produced).
3. **Required logic**: a field with `.default()` is absent from `required`; one with `.optional()` is too.
4. **(Optional) cross-deserialize**: pipe the output through Python `_deserialize_dspy_signatures`
   in a `core/modaic` test and assert `.equals` an equivalent hand-written `dspy.Signature`.

## Step order

1. Widen `ConfigField` / `ConfigJson` types in `config.ts` (add `$defs?` to the signature object).
2. Add `src/signatures/types.ts`: `Image`, `Audio`, `Scale`, `Enum` factories (Zod + `__modaic_type` meta); export from `index.ts`.

4. Implement `serializeSignatureToConfig` (assemble → sort → `{model:null, signature}`).
5. Add `tests/serialization/config.test.ts` (golden + type coverage + required + Scale/Enum/Image goldens).
6. `bun test`; (optional) add the Python cross-deserialize check (incl. a Scale/Enum/Image signature).

**No changes to `core/modaic` (Python SDK) or the server are required** — Scale/Enum serialize as
literals and Image/Audio reuse the existing `dspy.*` custom-type registry.
